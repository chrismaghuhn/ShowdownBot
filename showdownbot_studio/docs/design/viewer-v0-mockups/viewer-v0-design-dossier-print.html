ShowdownBot Studio — Phase 0
Replay + DecisionTrace Viewer

Visuelles und interaktives Design für eine ernsthafte Desktop-Analyse­anwendung. Godot 4.5.2 · typed GDScript · read-only versionierte Viewer-DTOs. Diese Datei ist reines Entwurfsmaterial und erklärt keinen Entwurf als freigegeben oder implementierungsbereit.

01 · Grundlage

Verbindliche Anforderungen & Non-Goals

Ableitung ausschließlich aus den fünf Quelldokumenten. Eine Anforderung ohne Quellenbeleg wurde nicht aufgenommen. Quellen-Chips verweisen auf viewer-v0-design.md (V), MASTER_SPEC.md (M), PROJECT_BOUNDARIES.md (B), ADR-001 (A).

Kernaufgabe

V§0,§3
  • ▹Ein eingefrorenes, deterministisches Bundle offline laden (immer nur eines).
  • ▹Kampf Zug für Zug replayen; aufgezeichnete Entscheidungen erklären, nicht neu berechnen.
  • ▹Direktstart per --decision battle_id:decision_index.

Abstraktes Board

V§6.1 · M§5
  • ▹Sprite-unabhängig: Namen, HP, Status, aktive Slots, Wetter/Terrain/Side-Conditions, Turn-State, eigene semantische Typ-Icons.
  • ▹Information-State visuell trennbar: known / suspected / unknown / not recorded.
  • ▹Muss ohne jedes optionale Artwork vollständig verständlich bleiben.

Timeline

V§5.3,§6.4 · A
  • ▹Turn- und Decision-Navigation; klare Trennung Battle Events ↔ Bot Decisions.
  • ▹Marker für knappe Entscheidungen, Warnungen, Fallbacks, degradierte Zustände.
  • ▹Bounded rendering — keine unbegrenzte Node-Liste (ADR: Godot ohne Virtual-List).

Decision & Candidates

V§6.5 · M§6
  • ▹Gewählter Kandidat über strukturelle candidate_key, nie über Tabellenposition.
  • ▹Aggregat-Score immer mit Aggregation-Mode; fehlt der Mode → sichtbar degradiert, nie aus config_hash abgeleitet.
  • ▹Deterministisches Sortieren/Filtern; Compact + Comfortable.

Fail-closed

V§7 · M§2.5
  • ▹Unbekannte Major-Version, Hash-Mismatch, doppelter decision_index, fehlende Pflichtdaten → verweigern, nicht raten.
  • ▹Keine erfundenen Ersatzwerte, keine stillen Defaults, keine scheinbar gültige Analyse.
  • ▹Fehlende Optional-Daten zeigen not recorded.

Zugänglichkeit & Desktop

M§2.4 · V§6.2,§9.2
  • ▹Verbindlich: Keyboard-first, sichtbarer Fokus, Kontrast, Status nie nur über Farbe, Skalierung 75–200%.
  • ▹Resizable/kollabierbare Panels, Mindestfenstergröße, Mixed-DPI, Reset-Layout.
  • ▹Screen-Reader nur best effort (AccessKit experimentell) — ehrlich zu berichten.

Non-Goals — in Phase 0 nicht entworfen

Live-Spectator Login / Account / Ladder Chat / Rooms Teambuilder Matchup-Analyzer Usage-Stats-Browser Damage Calculator Plugins / Mods Externe Bots Netzwerk / WebSocket Replay-Suche / Download Score-über-Zeit-Graph (→ v0.1) Sprite-Pakete Mobile / Browser / Controller Recompute / Re-Ranking Stärke-/Korrektheitsaussage

Spätere Phasen erscheinen im Phase-0-Entwurf gar nicht — nicht als deaktivierte Navigationseinträge. Der Screen bleibt auf die eine Aufgabe fokussiert.

DESIGN INPUT MISSING requires schema review
  • Latenz pro Entscheidung — Inspector wünscht sie, im Bundle-Schema nicht vorgesehen.
  • Fallback-Grund (Klartext) — nur fallback_used (Status) belegt, kein Reason-String.
  • State-Summary-Felder — „recorded state-summary“ ist nicht enumeriert (Boosts, Hazards, Slot-Detail).
  • Belief-/Hypothesis-Schema — „when present“, aber Felder/Wahrscheinlichkeiten unspezifiziert.

  • „suspected“-Quelle — Board unterscheidet 4 Info-States; die Herleitung von suspected aus Beliefs ist nicht definiert.

★

Empfehlung — Ansatz A: Analyse-Workbench

Nur die Workbench zeigt Battle-State, gewählte Aktion, Kandidaten-Scores und Provenance gleichzeitig — genau die Kette, die ein Diagnose- oder Validierungslauf ununterbrochen braucht. Sie ist zudem das in V§6.1 beschriebene Default-Layout. Die einzige echte Schwäche — hohe Dichte — ist durch die ohnehin verbindlichen Mittel Compact/Comfortable, kollabierbare Docks und resizable Panels beherrschbar. B und C werden nicht verworfen, sondern leben als Dock-Presets derselben Workbench weiter: „Replay-Fokus“ maximiert das Board, „Vergleich“ maximiert die Tabelle. Ein Reset-Layout stellt A wieder her.

Der wichtigste Nicht-Happy-Path: Fallback benutzt, aggregation.mode = null → Score als aggregation mode not recorded gekennzeichnet, optionale Telemetrie fehlt. Degradation ist prominenter als gewöhnlicher State. Keine erfundenen Ersatzwerte.

07 · Mockup

Bundle-Fehler — fail-closed Validation Gate

Kein Bundle wird still geöffnet. Primärfall: Hash-Mismatch — die betroffene Datei wird benannt, Trusted-Mode verweigert. Darunter drei weitere fail-closed Ausgänge. Kein „trotzdem öffnen“, keine Ersatzwerte.

BUNDLE INVALID Hash-Mismatch unsupported major malformed manifest load failed / cancelled
08 · Mockup

Compact Density

Dieselbe Workbench, dieselbe Information und Auswahl — nur dichter. Zeilenhöhen und Innenabstände schrumpfen, alle 12 Kandidaten sind ohne Scrollen sichtbar. Nichts wird entfernt, nur verdichtet (V§6.2, §9.2).
