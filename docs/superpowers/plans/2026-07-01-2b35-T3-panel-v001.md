## Cross-cutting rules
- **T3-CC-1 — held-out is mechanically separated + never tuned.** `Panel` keeps `dev_teams` and
  `heldout_teams` as **separate** lists. Dev schedule generation uses ONLY dev teams; held-out generation
  requires an explicit `confirm_heldout=True` and is refused otherwise. A test proves a dev schedule
  never contains a held-out team. Discipline (inherits parent CC-5): **no hyperparameter is ever read
  from held-out results** — only at the final gate (T6).
- **T3-CC-2 — every team is legal + CONTENT-hashed (Fix 1).** Each team `.txt` passes
  `validate-team gen9vgc2025regi` (exit 0) and has a `pack-team`-generated `.packed`. Each `PanelTeam`
  carries a **`team_hash` = `sha1(canonical({team_txt_content, packed_content}))[:16]`** (content, not
  just path). `panel_hash` includes version + policy list + dev/held-out split + per-team
  `{team_id, archetype, team_path, team_hash}`. **Gate:** editing a team file **without** changing its
  path MUST change `panel_hash`.
- **T3-CC-3 — reproducible policies are deterministic; non-reproducible excluded by default (Fix 3).**
  Paired comparison (T1b) needs deterministic bots. `heuristic`, `greedy_protect`, `simple_heuristic`,
  `scripted_vgc`, and the eval-only deterministic `max_damage` are reproducible. `random` is
  `reproducible=False` (registry, Fix 2). **`generate_dev_schedule` includes only `reproducible=True`
  policies by default; `random` (or any non-reproducible policy) requires `allow_nonreproducible=True`,
  and the generated schedule + report are marked `reproducible=False`.** So `random` can never
  accidentally enter a paired McNemar run.
- **T3-CC-4 — provenance travels; `panel_hash` reaches T2 rows (Fix 6).** Generated schedules carry
  `panel_hash` as schedule metadata; `run_schedule` writes that `panel_hash` into each T2 result row
  (`row.panel_hash`). Legacy schedules without `panel_hash` keep `row.panel_hash = null`. `schedule_hash`
  also travels (already in T1c/T2).
